# QMT 执行器部署方案（2026-07-22 调研, 电话前必读）

**硬事实**: miniQMT/xtquant 是 Windows-only(启动器是 .exe, 且要求 QMT 客户端
保持登录)。我们的主机是 macOS——这不挡开户, 但决定了架构和一笔小成本。

## 三个部署选项

| 方案 | 形态 | 成本 | 判断 |
|---|---|---|---|
| **A. Windows 云服务器全栈**(推荐) | 轻量云 Win Server 挂 QMT 客户端 + 执行器 python;Mac 只产目标仓位 | 2核4G ≈ ¥40-90/月(阿里/腾讯轻量) | 7×24 稳定, 不受家庭断网/合盖影响;量化挂机的社区标准做法 |
| B. Windows VPS + bridge, Mac 主控 | VPS 只跑 QMT+HTTP 桥(如开源 qmt-bridge), 执行逻辑留 Mac | 同上 + 桥的运维 | 多一层网络暴露面, 桥是第三方代码需审计;暂不选 |
| C. 闲置 Windows PC | 家里挂一台 | 电费 | 没有就不考虑 |

## 推荐架构(方案 A + git 作传输)

```
Mac(现有九本账/研究/观测层)
  └─ 目标仓位 targets.json ──push──▶ 私有仓 qtrade-records
                                        │ pull(分钟级轮询)
Windows VPS(QMT 客户端常登 + 执行器)
  └─ xtquant 下单(五层安全照 OKX broker 复制) ──fills──▶ push 回私有仓 → Mac 端 TCA 对账
```

- **git 当消息总线**: 无暴露端口、天然审计痕迹、月频/日频延迟完全够用;
  凭据只活在 VPS 的 QMT 客户端里(与 IBC 同款处理, 我不碰)
- 执行器 dry-run 默认, 真钱开关维持"模拟盘≥30天+过门槛+用户逐项确认"

## 电话时要多问的三件事(在原三件之外)

1. **QMT 权限形式**: 给的是 miniQMT(xtquant 库)还是完整版 QMT? 有无**模拟仓**
   (我们的 30 天前置要靠它)?
2. **登录环境限制**: 云服务器/异地 IP 登录 QMT 有无风控限制或报备要求?
3. **债券佣金**(E67 复活钥匙): 转债佣金能否到万 0.5 级? 顺带确认转债是否也走
   miniQMT 通道

## Mac 侧已预建完成(2026-07-24, 等权限即插即用)

代码在 `qtrade/live/qmt_exec.py`, 21 个单测全部在 macOS 上跑通:

- **总线消息** `qmt/targets_<book>.json`(私有仓, 每日随备份推送): schema 版本
  + sha256 校验和(截断的 git pull 不会变成交易单)+ 7 天新鲜度守卫(生产端
  死了消费端拒动)+ 做空拒收。消息带纸面账**股数与权益基数**而非权重——
  账本在决策日已经解过权重→股数, 执行器只做资金比例缩放, 不做第二次定价。
- **生产端已在跑**: `cli qmt-targets` 挂进 paper_all 每日备份闸门, 两本 CN 账
  (ashare_ml 50 持仓 / cb_double_low 20 持仓)的消息已在私有仓里每天刷新。
- **五层安全**照 OKX broker 移植: 账户钉扎(fail-closed)→ 订单级上限 →
  对账 RECONCILE(复用同一段已验证纯逻辑)→ 长多/单票/gross 钳制 → 回撤熔断。
  A股物理另加: T+1 可卖量、一手 100 股(清仓可零股)、停牌/涨跌停跳过重试。
  熔断只写 HALTED **不自动清仓**——T+1 下恐慌性自动平仓是不可逆的, 平仓是人的决定。
- **XtBroker**(xtquant 薄适配)已写好但**未经实测**——Windows-only, 激活日
  第一件事就是拿券商模拟仓把它的每个方法怀疑一遍。

权限下来当天: 你开 VPS(10 分钟)装 QMT 客户端登录 → 拉私有仓 → dry-run 对接
→ 模拟盘 30 天倒计时开始。云服务器那 ¥40-90/月是本项目第二笔真实开销
(第一笔是 LLM API ~$1/月)。
